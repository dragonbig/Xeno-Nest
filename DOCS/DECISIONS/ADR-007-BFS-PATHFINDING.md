# ADR-007: BFS 거리맵 기반 길찾기 도입

> **카테고리:** DECISIONS
> **최초 작성:** 2026-03-23
> **최종 갱신:** 2026-03-23
> **관련 기능:** 적 이동 시스템, 건물 시스템

**상태:** Accepted
**날짜:** 2026-03-23

---

## 맥락

ADR-001에서 BFS 경로탐색을 폐기하고 직선 추적(Direct Pursuit) + `findBuildingOnPath()` 방식을 채택했다. ADR-006에서 맵을 30x24로 확대하고 불규칙 타원형 외벽을 도입한 뒤, 직선 추적에 3단계 슬라이딩 로직을 추가하여 외벽 통과를 방지했다.

그러나 슬라이딩 방식에는 근본적 한계가 있었다:

1. **우회 경로 불가:** 슬라이딩은 벽에 부딪히면 X/Y 축 분리 이동만 시도한다. 외벽 형태에 따라 적이 벽 모서리에 끼여 진행하지 못하는 사각지대가 발생했다.
2. **findBuildingOnPath의 부정확성:** 직선 투영 기반 건물 탐지는 적이 직선 경로가 아닌 슬라이딩으로 우회할 때 실제 경로 상에 없는 건물을 타겟으로 잡거나, 경로 상의 건물을 놓치는 문제가 있었다.
3. **4방향 스폰 도입:** 스폰 지점이 3시/5시/7시/9시 4방향으로 배치되면서 적이 다양한 각도로 기지에 접근해야 하며, 직선 추적으로는 복잡한 외벽 형태를 따라 입구까지 도달하기 어려웠다.

---

## 결정

직선 추적(Direct Pursuit) + `findBuildingOnPath()` 방식을 전면 폐기하고, NEST에서 역방향 multi-source BFS로 계산한 거리맵(distance map)을 따라 이동하는 방식으로 전환한다.

### 핵심 구조

1. **`computeDistanceMap()`**: NEST가 점유하는 모든 타일(2x2 = 4타일)을 시작점(거리 0)으로, 4방향 BFS를 수행하여 각 타일까지의 거리를 계산한다.
   - `BLOCKED` 타일: 통행 불가 (BFS 확장하지 않음)
   - 건물 타일(WALL, THORN, SPORE, REPAIR_BLD, RESOURCE): 거리는 기록하되 큐에 넣지 않음 (적이 건물 위치까지는 접근하지만 건물 너머로 경로가 이어지지 않음)
   - `EMPTY`, `ENTRANCE`, `NEST`: 통과 가능 (BFS 확장)

2. **`pursueTarget()` 전면 재작성**: 적은 인접 4칸의 거리맵 값을 비교하여 가장 작은 방향으로 이동한다. 인접 타일에 건물이 있으면 해당 건물을 타겟으로 설정하고 공격한다.

3. **`moveToward(e, tx, ty, dt)` 이동 헬퍼 분리**: 슬로우 적용, BLOCKED 충돌(`hitsBlockedTile`), 축별 슬라이딩 로직을 하나의 함수로 통합했다.

4. **`G.distanceMap` / `G.distanceMapDirty` 상태**: 건물 배치(`createBuilding`), 건물 제거(`removeBuilding`), 건설 완료 시 `distanceMapDirty = true`로 설정하여 다음 프레임 시작 시 재계산을 트리거한다.

5. **`findBuildingOnPath()` 완전 제거**: 직선 투영 기반 건물 탐지 함수를 삭제했다. 건물 탐지는 `pursueTarget()` 내부에서 인접 타일 건물 탐색으로 대체되었다.

---

## 이유

### BFS 거리맵은 모든 위치에서 최적 이동 방향을 제공한다

거리맵이 한 번 계산되면, 임의의 위치에서 NEST까지의 최단 경로 방향을 O(1)로 조회할 수 있다. 적의 수에 관계없이 동일한 거리맵을 공유하므로, 180마리의 적이 동시에 이동해도 추가 탐색 비용이 없다.

### 건물이 경로를 자연스럽게 차단한다

건물 타일은 거리를 기록하되 큐에 넣지 않으므로, 건물 배치 시 해당 타일 너머의 경로가 자동으로 차단된다. 성벽을 여러 겹으로 배치하면 적이 첫 번째 성벽까지만 접근하고, 성벽을 파괴해야만 내부로 진행할 수 있다. 이는 직선 추적에서의 `findBuildingOnPath()`보다 정확하다.

### ADR-001의 성능 우려는 해소되었다

ADR-001에서 BFS를 기각한 이유 중 하나는 "건물 배치/파괴마다 192개 노드 탐색" 비용이었다. 현재 맵은 30x24 = 720 타일이지만, BFS는 건물 변경 시에만 1회 수행되고 결과를 `G.distanceMap`에 캐싱한다. 웨이브 중 매 프레임 호출되지 않으며, 720개 노드 BFS는 JavaScript에서 1ms 미만으로 완료된다.

### 4방향 스폰에 자동 대응

거리맵은 맵 전체를 커버하므로, 3시/5시/7시/9시 어느 방향에서 스폰하든 적이 자동으로 NEST까지의 최단 경로를 따라 이동한다. 직선 추적처럼 스폰 방향별 특수 처리가 필요하지 않다.

## 검토한 대안

### 대안 A: 직선 추적 유지 + 슬라이딩 개선

- ADR-006에서 도입한 3단계 슬라이딩을 더 정교하게 개선하는 방식.
- 장점: 기존 코드 변경 최소화.
- 기각 이유: 불규칙 외벽과 4방향 스폰 조합에서 슬라이딩만으로는 모든 경로를 처리할 수 없었다. 사각지대가 근본적으로 해결되지 않았다.

### 대안 B: A* 경로탐색 (적별 개별 계산)

- 각 적이 자신의 위치에서 NEST까지 A* 경로를 계산하는 방식.
- 장점: 적별로 최적 경로를 제공.
- 기각 이유: 최대 180마리의 적이 매 건물 변경 시마다 개별 A*를 수행하면 성능 부담이 크다. BFS 거리맵은 1회 계산으로 모든 적에게 공유되므로 훨씬 효율적이다.

### 대안 C: Flow Field (연속 벡터 필드)

- 타일별로 이동 방향 벡터를 저장하는 방식.
- 장점: 부드러운 이동, 타일 단위가 아닌 연속 방향 제공.
- 기각 이유: 구현 복잡도가 높고, 타일 기반 BFS 경사 추적으로도 충분히 자연스러운 이동이 가능했다.

---

## 결과

- `findBuildingOnPath()` 함수가 완전히 제거되었다.
- `pursueTarget()`이 거리맵 경사 추적 + 인접 건물 탐지 방식으로 전면 재작성되었다.
- `moveToward(e, tx, ty, dt)` 이동 헬퍼가 분리되어, 슬로우/BLOCKED 충돌/슬라이딩을 일관되게 처리한다.
- `G.distanceMap`(Float32Array[][])과 `G.distanceMapDirty`(boolean)이 게임 상태에 추가되었다.
- `createBuilding`, `removeBuilding`, 건설 완료 시 `distanceMapDirty = true`가 설정된다.
- 매 프레임 시작 시 `distanceMapDirty === true`이면 `computeDistanceMap()`이 호출된다.
- ADR-001의 상태가 "Superseded by ADR-007"로 변경되었다.
- ADR-006의 "대안 C: BFS 경로탐색으로 전환" 기각 이유가 이 결정으로 번복되었다. 당시에는 슬라이딩으로 충분하다고 판단했으나, 4방향 스폰 도입 후 BFS 전환이 불가피해졌다.

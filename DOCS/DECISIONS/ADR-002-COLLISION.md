# ADR-002: O(n²) 충돌 → 공간 해시 기반 충돌 전환

> **카테고리:** DECISIONS
> **최초 작성:** 2026-03-23
> **최종 갱신:** 2026-03-23
> **관련 기능:** 적 충돌 처리, resolveCapsuleCollisions

**상태:** Accepted
**날짜:** 2026-03-23

---

## 맥락

적끼리 겹치지 않도록 매 프레임 충돌을 해소해야 한다. 나이브하게 구현하면 모든 적 쌍을 비교하는 O(n²) 방식이 된다.

최대 적 수는 ENEMY_CAP 합산 기준으로 이론상 최대 280마리(CITIZEN 80 + SCOUT 100 + ... )이지만, 실제 게임 중 WAVE 후반 최대치에서 약 180마리 수준이다. 180마리 기준 O(n²) 검사 쌍 수:
```
180 × 179 / 2 = 16,110 쌍/프레임
```

---

## 결정

`resolveCapsuleCollisions()`에서 `TILE_SIZE(48px)` 크기의 공간 해시 그리드를 사용한다. 각 적을 해당 셀에 등록하고, 자신의 셀 + 인접 8방향 셀(3×3)에 속한 적끼리만 충돌을 검사한다.

---

## 이유

### 정확성 보장

게임에서 충돌 가능한 최대 반지름 쌍은 FAST(12) + FAST(12) = 24px이고, 1px 여유를 더하면 25px이다. 셀 크기 48px > 25px이므로, 충돌 가능한 두 적은 반드시 같은 셀 또는 인접 셀에 존재한다. 누락 없음이 수학적으로 보장된다.

```
최대 충돌 거리: 15 + 15 = 30px (TANKER + TANKER)
셀 크기: 48px > 30px  → 인접 1칸 이내 검사로 충분
```

### 성능 향상

```
O(n²): ~16,110 쌍/프레임
공간 해시: ~1,600 쌍/프레임 (약 10배 감소, game.js 주석 명시)
```

120fps 기준 16,110쌍 × 120 = 약 193만 쌍/초 → 1,600쌍 × 120 = 약 19만 쌍/초로 감소.

### 구현 단순성

매 프레임 그리드를 새로 생성(`new Map()`)하므로 이전 프레임 상태를 정리할 필요가 없다. 해시 키는 `cx * 10000 + cy` 정수 연산으로 충분하다(12×16 맵 기준 최대 키 값 12×10000+16=120016으로 충돌 없음).

---

## 검토한 대안

### O(n²) 브루트 포스

- 장점: 구현 0줄 추가, 항상 정확
- 단점: 적 수가 많아질수록 프레임 드롭 발생. 모바일에서 특히 문제.

### Quadtree

- 장점: 동적 공간 분할, 밀집 분포에 효과적
- 단점: 구현 복잡도 높음. 12×16 고정 맵에서 공간 해시보다 이점 없음.

---

## 결과

- `resolveCapsuleCollisions()`는 매 프레임 호출되며, 공간 해시를 즉석 생성 후 사용하고 버린다
- 적 최대 밀집 시나리오(~180마리)에서도 충돌 처리가 프레임 버짓의 주요 병목이 되지 않는다
- 쌍 중복 처리 방지는 `other.id > e.id` 조건으로 보장한다

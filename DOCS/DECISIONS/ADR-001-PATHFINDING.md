# ADR-001: BFS 경로탐색 → 직선 추적 전환

> **카테고리:** DECISIONS
> **최초 작성:** 2026-03-23
> **최종 갱신:** 2026-03-23
> **관련 기능:** 적 이동 시스템

**상태:** Superseded by ADR-007
**날짜:** 2026-03-23

> **폐기 사유:** ADR-007에서 BFS 거리맵 기반 길찾기로 전환되었다. 직선 추적(Direct Pursuit) 방식과 `findBuildingOnPath()` 함수는 완전히 제거되었다. 적은 이제 NEST에서 역방향으로 계산된 거리맵의 경사를 따라 이동하며, 경로 상 건물을 인접 타일 탐색으로 자동 탐지한다.

---

## 맥락

적 AI가 핵심 둥지까지 경로를 탐색하는 방식을 결정해야 했다. 초기 설계(game.js 상단 주석)에는 BFS가 언급되어 있었다.

```
// - BFS는 건물 배치/파괴 때마다 호출되며, 결과(경로)를 G.path에 캐싱한다.
```

Phase 1 구현 과정에서 이 설계를 직선 추적으로 대체했다.

---

## 결정

BFS 기반 경로탐색을 폐기하고, 직선 추적(Direct Pursuit) + 장애물 즉시 타겟팅 방식으로 구현한다.

구체적으로:
- 적은 매 프레임 현재 위치 → 둥지를 잇는 직선을 계산한다
- 직선 위에 건물(성벽/타워/자원건물)이 있으면 그 건물이 즉시 타겟
- 없으면 핵심 둥지가 타겟
- 타겟이 파괴되면 다음 프레임에 자동으로 새 타겟 설정 (재경로 계산 없음)

---

## 이유

### 성능

BFS는 건물 배치/파괴 때마다 12×16 그리드를 탐색한다. 건물이 다수 배치된 상태에서 WAVE 중 건물이 파괴될 때마다 192개 노드를 탐색하고, 결과를 캐싱하더라도 100여 마리의 적이 동시에 캐싱된 경로를 참조하는 방식은 복잡한 동기화를 요구한다.

직선 추적은 `Math.hypot`과 내적 계산만으로 적당 O(B) (B = 건물 수)이며, 매 프레임 호출해도 부담이 없다.

### 단순성

12×16 소규모 맵에서 BFS의 핵심 이점인 '장애물 회피 경로'는 게임플레이상 필요하지 않다. 플레이어가 성벽으로 적을 막는 것이 핵심 전략이며, 적이 성벽을 우회하는 경로를 찾으면 성벽의 가치가 없어진다. 직선 추적은 적이 성벽에 정직하게 막히는 동작을 자연스럽게 만든다.

### 즉시 재타겟팅

BFS 경로 캐싱은 경로 위의 건물이 파괴될 때 캐시 무효화 + 재탐색이 필요하다. 직선 추적은 `G.path` 캐시가 없으므로 재경로 계산 트리거 자체가 불필요하다.

---

## 검토한 대안

### BFS 경로탐색 + G.path 캐싱

- 장점: 성벽 배치 방식에 관계없이 적이 항상 둥지까지 도달하는 경로를 찾음
- 단점: 건물 배치/파괴마다 재탐색 비용, 경로 공유 시 동기화 복잡성, 성벽 전략의 유효성 감소

### 스티어링 행동(Steering Behaviors)

- 장점: 부드러운 이동, 자연스러운 군집 행동
- 단점: 구현 복잡도 높음, 이 규모의 게임에서 과설계

---

## 결과

- `G.path` 필드가 G 초기화 구조에 존재하지 않는다
- 모든 적 이동은 `pursueTarget()`과 `findBuildingOnPath()`로 처리된다
- 성벽이 직선 경로를 막으면 적은 성벽을 공격하고, 파괴된 후에야 다음 타겟으로 이동한다
- 성벽을 우회하는 적 행동은 의도적으로 구현하지 않는다
